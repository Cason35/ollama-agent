# Day5 学习总结：JSON 结构化输出（Agent 核心能力）

## 今日任务做了什么

今天在 `ollama-chat-day5` 项目里，完成了从“普通聊天”到“可被程序理解的结构化输出”的升级，核心是让模型输出 JSON，再由后端解析并分流。

### 1. 新建 Day5 项目并迁移业务代码

- 在 `ollama-chat-day5` 下重新搭建了一个 Next.js 项目（不是整包复制 day4）
- 迁移了 day4 的聊天业务基础（前端聊天页 + 后端 `/api/chat` 路由）
- 保留了基础聊天 UI 和请求流程，作为 Day5 的改造起点

### 2. 后端加入 system prompt，强约束 JSON 输出

在 `app/api/chat/route.ts` 中新增系统提示词，要求模型：

- 必须严格输出 JSON
- 不允许输出额外解释文字
- 使用固定结构：

```json
{
  "action": "chat | search",
  "content": "回答内容",
  "keyword": "如果是search，这里是关键词"
}
```

并在请求 Ollama 时将 system 消息放入 `messages` 首位。

### 3. 后端实现 JSON 解析 + fallback 兜底

为避免模型不听话导致程序崩溃，增加了两层容错：

- 第一层：直接 `JSON.parse(modelOutput)`
- 第二层：如果失败，用正则提取 JSON 主体再 parse：
  - `modelOutput.match(/\{[\s\S]*\}/)`
- 最终兜底：若仍失败，回退为：
  - `action: "chat"`
  - `content: 原始输出`
  - `keyword: ""`

### 4. 后端根据 action 分流（Agent 雏形）

在路由中根据 `parsed.action` 返回不同类型：

- `search` -> 返回 `{ type: "search", keyword }`
- `chat` -> 返回 `{ type: "chat", content }`

这一步本质上就是“决策能力”的第一版：模型输出意图，程序执行分支。

### 5. 前端根据返回 type 展示不同结果

在 `app/page.tsx` 中实现类型分支渲染：

- `type === "search"` 显示：`🔍 搜索：关键词`
- `type === "chat"` 显示普通助手回复

用户能直观看到系统已经不是单一聊天，而是开始有“行为类型”了。

## 今天学到了什么

## 1) 为什么 JSON 输出是 Agent 的基础

以前模型只会“自然语言回答”，程序很难稳定处理；
现在模型输出结构化字段（如 `action`、`keyword`），程序就能像执行指令一样运行流程。

一句话：**从“会聊天”升级到“可编排”**。

## 2) Prompt 约束很重要，但永远不够

即使写了“必须输出 JSON”，模型仍可能输出：

- “好的，以下是 JSON：...”
- 前后多一段说明文字

所以工程里必须做“解析容错”，不能把稳定性赌在 prompt 上。

## 3) 工程化思维：永远给失败留后路

关键实践是：

- `try/catch` 防止解析报错导致接口崩溃
- 正则提取 JSON，处理“半听话”输出
- fallback 保证接口始终返回可用结构

这比“理想情况下能跑”更重要。

## 4) Agent 的最小闭环

今天已经形成一个最小闭环：

1. 用户输入
2. 模型输出结构化 JSON（意图）
3. 后端解析并分流
4. 前端按行为展示

这就是后续接真实工具（搜索 API、数据库、函数调用）的基础模板。

## 今日验收结论

目标状态已经具备：

- 普通问候 -> `chat`
- 查询型意图（如天气）-> `search`
- 其他闲聊（如笑话）-> `chat`

Day5 的核心价值是：**把模型输出变成“程序可执行的决策信号”**，为真正的 Agent 能力打下了地基。
