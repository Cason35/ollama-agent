# 最新技术路线（Ollama 版）
## 当前主栈
- 前端：Next.js + React + TypeScript
- 模型服务：Ollama
- 主模型：`qwen2.5:14b`
- 后续可选模型：`qwen2.5-coder:14b`
- 本地接口：
  - Ollama 原生：`http://localhost:11434/api/chat`
  - OpenAI 兼容：`http://localhost:11434/v1`

## 为什么先用 Ollama
1. 本地可跑，不需要先充值 API
2. 足够学习 Agent 基础
3. 可以练前端产品能力和工程能力
4. 后面如果切换到 OpenAI API，很多代码和思路可以复用

## 要注意的限制
Ollama 的 OpenAI 兼容 `responses` 接口当前是**非状态化**的，因此：
- 不支持 `previous_response_id`
- 聊天历史需要自己维护
- 一些云端平台内置工具和 trace 能力暂时没有

这不影响前 1-2 个月的学习。

#### 第 10 天学习任务（重点：把 Demo 变成可用系统）
##### 第 10 天核心目标（一句话）
让 `summary` / `todo` 从占位功能升级为可复用的真实 Agent 能力，并让系统具备可观测、可调试、可稳定的工程基础。

##### 任务 1：`summary` 真实化
- 定义 `SummaryAction` schema，明确输入契约（`action` + `content`）
- 从 `messages` 中裁剪最近上下文（建议 `slice(-6)`）避免 token 膨胀
- 构造强约束总结 prompt（3-5 条要点 + 关键信息 + 简洁表达）
- 调用 Ollama `/api/chat` 生成总结
- 返回统一结构（如 `type: "summary"` + `data`）

##### 任务 2：`todo` 真实化
- 定义 `TodoItem` 结构（`task` + `done`）
- 使用“只输出 JSON 数组”的强约束 prompt
- 将模型输出做 `JSON.parse`，并补齐解析失败 fallback
- 输出至少 3 条可执行待办，确保前端可直接渲染列表

##### 任务 3：可观测性建设
- 每次工具调用记录 `action` 与输入
- 记录耗时（`Date.now()` 起止）
- 完整捕获并输出错误日志（便于定位失败点）
- 保证一次请求能看到：调用了什么、输入是什么、耗时多少、成功还是失败

##### 任务 4：关键测试用例
- 正常 case：`帮我总结一下刚刚内容`
- 边界 case：空输入、短输入
- 异常 case：`todo` JSON 解析失败
- 验收目标：不崩溃、有 fallback、行为可预期

##### 第 10 天验收清单
- ✅ summary 可真实调用模型并返回 3-5 条要点
- ✅ todo 返回 JSON 且可被前端稳定渲染
- ✅ 工具调用日志与耗时可见
- ✅ 关键正常/边界/异常 case 已覆盖
