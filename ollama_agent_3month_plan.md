# 最新技术路线（Ollama 版）
## 当前主栈
- 前端：Next.js + React + TypeScript
- 模型服务：Ollama
- 主模型：`qwen2.5:14b`
- 后续可选模型：`qwen2.5-coder:14b`
- 本地接口：
  - Ollama 原生：`http://localhost:11434/api/chat`
  - OpenAI 兼容：`http://localhost:11434/v1`

## 为什么先用 Ollama
1. 本地可跑，不需要先充值 API
2. 足够学习 Agent 基础
3. 可以练前端产品能力和工程能力
4. 后面如果切换到 OpenAI API，很多代码和思路可以复用

## 要注意的限制
Ollama 的 OpenAI 兼容 `responses` 接口当前是**非状态化**的，因此：
- 不支持 `previous_response_id`
- 聊天历史需要自己维护
- 一些云端平台内置工具和 trace 能力暂时没有

这不影响前 1-2 个月的学习。

##### 第 4 天关键收获
1. **Streaming 本质**
   - 后端返回流（`response.body`）
   - 前端逐块读取（`reader.read()`）
   - UI 持续更新（逐步拼接 assistant 内容）
   - 核心认知：这是“边生成边渲染”的链路协同，不只是前端动画效果。
2. **卡顿问题根因**
   - `messages` 历史持续增长
   - 每次请求携带全部上下文
   - 前端高频 `setState`
   - 这是典型的 **Context Explosion（上下文膨胀）** 问题。

##### 第 4 天当前系统能力

- ❌ 无结构化输出（JSON）
- ❌ 无工具调用（Tool Calling）

#### 第 5 天任务（下一步）
##### 第 5 天主题：JSON 结构化输出（进入 Agent 核心）

##### 第 5 天为什么要做
当前模型主要是“自然语言聊天输出”，程序难以稳定解析。  
目标是让模型“按规则输出”，从而支持动作分发、工具调用和流程编排。

##### 第 5 天核心任务
1. **让模型输出 JSON**
   ```json
   {
     "action": "search",
     "keyword": "北京天气"
   }
   ```
2. **后端解析 JSON**
   ```ts
   JSON.parse(modelOutput)
   ```
3. **前端按 JSON 执行逻辑**
   - `action=search` -> 调用搜索逻辑
   - `action=chat` -> 回落普通聊天展示

##### 第 5 天验收标准
- [ ] 模型能稳定输出 JSON
- [ ] 后端成功解析 JSON
- [ ] 前端根据 JSON 执行动作
- [ ] 全链路无解析错误

##### 第 5 天可选优化
- `try/catch` JSON 校验
- 解析失败兜底（fallback 到普通聊天）
- prompt 优化（提升结构化输出稳定性）
