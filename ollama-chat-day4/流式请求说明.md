# Ollama Chat Day4 流式请求说明

本文基于以下两个文件：

- `ollama-chat-day4/app/page.tsx`
- `ollama-chat-day4/app/api/chat/route.ts`

目标是讲清楚三件事：

1. 这套流式请求整体怎么跑
2. 后端要做什么处理
3. 前端要做什么处理，以及前后端如何对齐

---

## 1. 整体链路（从输入到流式显示）

一次对话的完整流程如下：

1. 用户在前端输入问题，点击发送。
2. 前端先把用户消息加入本地消息列表，同时加一条空的 assistant 占位消息（用于后续逐步填充）。
3. 前端调用同源接口 `POST /api/chat`，只传 `messages`。
4. Next.js 后端路由接到请求后，做参数校验。
5. 校验通过后，后端再请求本机 Ollama：`POST http://localhost:11434/api/chat`，并固定 `stream: true`。
6. Ollama 返回 NDJSON 字节流（每行一个 JSON 增量片段）。
7. 后端不做全文拼装，直接把这个流透传给浏览器。
8. 前端用 `ReadableStream` + `TextDecoder` 持续读取 chunk，按“按行”解析 NDJSON。
9. 每解析出一段 `message.content` 增量，就拼到 `fullText`，并更新最后一条 assistant 消息。
10. 用户看到打字机式实时输出；流结束后进行最后一次 flush，避免尾字丢失。

---

## 2. 后端要做什么（`app/api/chat/route.ts`）

后端在这里的定位是“业务代理层”，不是模型推理层。核心职责有 4 个：

### 2.1 请求入口统一与参数兜底

- 统一入口：浏览器只打 `/api/chat`，不直接访问 Ollama。
- 参数校验：`messages` 必须是非空数组，否则直接 `400`，减少无效请求。

### 2.2 负责模型配置与策略收口

后端固定了模型和流式开关：

- `model: "qwen2.5:14b"`
- `stream: true`

这样前端无需关心模型名与底层策略，未来换模型、加鉴权、加限流都在这层改。

### 2.3 流式透传（关键）

请求 Ollama 成功后：

- 不等待完整回复，不把流读完再返回。
- 直接 `return new Response(res.body, ...)` 把可读流交给前端。
- 保留 `Content-Type`（默认回退 `application/x-ndjson`），并加 `Cache-Control: no-cache`。

这一步决定了“前端能否实时看到增量”。

### 2.4 错误归一化

后端将异常统一转为 JSON 错误：

- Ollama 非 2xx：返回 `{ error }`，HTTP 500。
- 响应 body 为空：返回 `{ error: "Empty stream from Ollama" }`。
- catch 场景（如 Ollama 未启动）：返回 `{ error: "Internal server error" }`。

这样前端只需按一种方式处理错误分支。

---

## 3. 前端要做什么（`app/page.tsx`）

前端在流式场景下，不只是“发请求然后拿结果”，而是要处理“持续到来的增量”。

### 3.1 发送前先准备 UI 状态

`handleSend()` 里做了两件很重要的事：

- 先把用户消息放进 `messages`。
- 立刻追加一条空 assistant 消息作为“流式写入目标”。

这让界面在请求发出瞬间就进入“对话继续中”的状态，不会卡住。

### 3.2 成功分支：消费 ReadableStream

当前端拿到 `res.ok` 后：

1. `res.body.getReader()` 获取流读取器。
2. `TextDecoder("utf-8")` 把 Uint8Array chunk 转字符串。
3. 用 `lineBuffer` 处理“半行”问题（一个 JSON 行可能被网络切成两段）。
4. 调用 `consumeOllamaNdjsonLines()` 按 `\n` 分行解析。
5. 从每行 JSON 提取 `message.content` 增量，追加到 `fullText`。
6. 每轮循环后 `setMessages` 更新最后一条 assistant，实现实时刷新。

### 3.3 为什么必须“按行缓冲 + 增量拼接”

Ollama 流是 NDJSON，不保证每次 `reader.read()` 就是完整 JSON 行。  
如果不做缓冲，直接 `JSON.parse(chunk)`，在跨 chunk 的场景下会大量报错并丢字。

当前实现用：

- `parts = buffer.split("\n")`
- `rest = parts.pop()`

把最后未完成的一行留到下一轮，能稳定处理网络分片。

### 3.4 结束与异常处理

- 流结束时调用一次 `decoder.decode()` flush 内部残留字符。
- 再跑一轮 NDJSON 消费，避免最后一小段没显示出来。
- `finally` 一定 `setLoading(false)`，避免按钮长期“生成中...”。
- `catch` 时把兜底文案写回最后一条 assistant，用户感知更一致。

## 4. 前后端连接点（接口契约）

这套流式方案可稳定运行，依赖前后端三个约定严格对齐：

### 4.1 请求契约（前端 -> 后端）

- URL：`POST /api/chat`
- Body：`{ messages: Array<{ role, content }> }`
- 前端不传模型、不传 stream，交给后端治理。

### 4.2 响应契约（后端 -> 前端）

- 成功：HTTP 2xx + 可读流 `res.body`（NDJSON）
- 失败：HTTP 非 2xx + JSON `{ error: string }`

前端代码正是按这两种分支处理：

- `!res.ok` -> `await res.json()` 读取错误
- `res.ok` -> 流式读取 `res.body`

### 4.3 数据格式契约（Ollama NDJSON）

前端默认每行 JSON 可能包含：

- `message.content`：本帧增量文本

所以前端只抽这一个字段，并不断拼接成完整回答。

---

## 5. 一句话记忆版

- 后端：**校验 + 转发 + 透传流 + 统一错误**
- 前端：**占位 + 读流 + 按行解析 NDJSON + 增量刷新 UI**
- 连接本质：**同一个接口的双分支协议（错误走 JSON，成功走流）**

---

## 6. 后续可扩展方向

1. 在后端增加超时控制和重试策略（防止 Ollama 卡住）。
2. 引入 `AbortController`，支持前端“停止生成”。
3. 给流式协议增加事件类型（如 `delta`、`done`、`error`），提高可观测性。
4. 在后端增加鉴权/限流，避免接口被滥用。
5. 记录对话与 token 统计，便于运维和成本分析。
