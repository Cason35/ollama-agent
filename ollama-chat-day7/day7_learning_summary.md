# 第7天学习总结：真实 Tool Calling（接入实时 API）

## 一、今天完成了什么

第7天的核心是把系统从「模型 + 假工具 + 假数据」升级到「模型 + 真实 API + 实时数据」，让 Agent 的工具调用能力真正可用。

本次已完成：

- 将 `fakeSearch` 升级为真实天气工具 `realWeather`
- 接入 Open-Meteo API，返回实时天气数据
- 保留 `chat / search` 分流逻辑，实现正常聊天与工具调用并存
- 优化关键词提取，让「帮我查北京天气」「北京的天气情况」都能稳定提取为「北京」
- 优化 Prompt，约束模型输出稳定 JSON，降低工具调用抖动
- 前端新增工具结果展示格式：`🔍 城市` + `📌 实时结果`

---

## 二、后端能力升级（`app/api/chat/route.ts`）

后端本次重点是「让模型输出可控」和「让工具执行可靠」。

### 1) 模型输出约束与解析增强

- 通过系统提示词强制模型输出 JSON（`action / content / keyword`）
- 增加 `parseModelOutput` 与 `normalizeParsedOutput`
- 支持两类恢复路径：
  - 模型输出是标准 JSON：直接解析
  - 模型夹带文本：从文本中提取 JSON 片段再解析
- 最终兜底：解析失败时回退为普通聊天，避免接口崩溃

### 2) 关键词提取增强

- 新增 `extractWeatherCity`，优先直接命中城市名（北京、上海）
- 对用户自然语言做清洗（去标点、去语气词、去「天气情况」等后缀）
- 清洗后再次匹配城市，提升命中率和稳定性

### 3) 真实天气工具 `realWeather`

- 内置城市经纬度映射（当前支持北京、上海）
- 调用 Open-Meteo 接口获取 `current_weather`
- 使用 `cache: "no-store"`，尽量保证结果实时
- 增加错误处理：
  - 城市不支持
  - 上游服务不可用
  - 返回字段缺失（如温度为空）
- 返回结果更完整：温度 + 可选风速

### 4) 执行链路

完整链路已形成：

1. 前端提交多轮消息上下文  
2. 模型输出结构化决策（`chat` 或 `search`）  
3. 后端按 `search` 分支执行 `realWeather`  
4. 返回工具响应（关键词 + 实时结果）  
5. 前端按类型渲染

## 三、前端体验升级（`app/page.tsx`）

前端重点是把工具调用结果「清晰地展示给用户」，并保证交互体验稳定。

- 统一 API 响应联合类型：`chat` 与 `search`
- 发送时采用乐观更新：先显示用户消息，再等待服务端
- `search` 结果使用两行紧凑样式展示：
  - `🔍 关键词`
  - `📌 工具结果`
- 增加错误提示：
  - 服务端错误展示为页面错误 + 助手气泡
  - 网络异常给出连接提示（如 Ollama 不可达）
- 自动滚动到底部，提升连续对话体验
