## 二、后端能力升级（`app/api/chat/route.ts`）

后端本次重点是「让模型输出可控」和「让工具执行可靠」。

### 1) 模型输出约束与解析增强

- 通过系统提示词强制模型输出 JSON（`action / content / keyword`）
- 增加 `parseModelOutput` 与 `normalizeParsedOutput`
- 支持两类恢复路径：
  - 模型输出是标准 JSON：直接解析
  - 模型夹带文本：从文本中提取 JSON 片段再解析
- 最终兜底：解析失败时回退为普通聊天，避免接口崩溃

### 2) 关键词提取增强

- 新增 `extractWeatherCity`，优先直接命中城市名（北京、上海）
- 对用户自然语言做清洗（去标点、去语气词、去「天气情况」等后缀）
- 清洗后再次匹配城市，提升命中率和稳定性

### 3) 真实天气工具 `realWeather`

- 内置城市经纬度映射（当前支持北京、上海）
- 调用 Open-Meteo 接口获取 `current_weather`
- 使用 `cache: "no-store"`，尽量保证结果实时
- 增加错误处理：
  - 城市不支持
  - 上游服务不可用
  - 返回字段缺失（如温度为空）
- 返回结果更完整：温度 + 可选风速

### 4) 执行链路

完整链路已形成：

1. 前端提交多轮消息上下文  
2. 模型输出结构化决策（`chat` 或 `search`）  
3. 后端按 `search` 分支执行 `realWeather`  
4. 返回工具响应（关键词 + 实时结果）  
5. 前端按类型渲染

## 七、第 8 天学习计划（重点：上下文裁剪）

### 今日主题：上下文裁剪（Context Window 管理）

一句话理解：

- ❌ 现在：每次都把全部聊天记录丢给模型
- ✅ 今天：只给“必要的信息”

### 你现在问题的本质

你当前代码大概率是：

```ts
messages: [
  ...所有历史消息
]
```

这会导致：

- 越聊越长
- token 越来越多
- 响应越来越慢
- 前端 `setState` 高频更新导致卡顿

### 核心概念（必须理解）

1. LLM 有上下文窗口上限（例如 8k / 32k token）
2. 不是“消息越多越好”，而是“信息越有用越好”

### 第 8 天任务拆解（一步一步来）

#### Step 1：最简单裁剪（先只保留最近 N 条）

后端先实现最低成本版本：

```ts
const MAX_MESSAGES = 10;
const trimmedMessages = messages.slice(-MAX_MESSAGES);

messages: [
  { role: "system", content: systemPrompt },
  ...trimmedMessages
]
```

验收点 1：

- 聊 50 轮不会明显变慢
- 请求体大小趋于稳定

#### Step 2：保留关键上下文（升级版）

问题：

- 如果用户前面说“我叫 Cason”
- 后面问“我叫什么？”
- 若关键消息被裁掉，记忆就会失效

简单可用策略：

- 保留 `system prompt`
- 保留最近 N 条
- 可选保留第一条用户关键信息

```ts
const trimmedMessages = [
  messages[0], // system
  ...messages.slice(-10)
];
```

#### Step 3（进阶，可选）：摘要记忆（Summary Memory）

思路：

- 旧对话压缩为一句总结
- 用总结替代旧消息，降低 token 占用

示例：

```ts
const summary = "用户叫Cason，喜欢AI";

messages = [
  { role: "system", content: systemPrompt },
  { role: "system", content: summary },
  ...recentMessages
];
```

验收点 2：

- 即使裁剪后，也能保留关键身份信息和偏好

#### Step 4：前端优化（解决流式卡顿）

你第 4 天已经遇到 `setState` 高频导致的卡顿，这里要做节流更新：

```ts
// ❌ 每个 chunk 都 set
setMessages();

// ✅ 节流更新
if (chunkCount % 3 === 0) {
  setMessages();
}
```

或使用：

```ts
requestAnimationFrame(() => {
  setMessages();
});
```

验收点 3：

- 流式输出不卡
- UI 更顺滑
